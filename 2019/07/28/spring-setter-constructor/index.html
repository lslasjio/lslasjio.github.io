Spring框架中的核心概念之一就是控制反转（IoC），或成为依赖注入（DI），本文中将简单介绍控制反转的概念，以及常用的两种依赖注入，讨论两者间的一些特点和区别。

控制反转（IoC）

Ioc—Inversion of Control，即“控制反转”，是Java开发中的一种设计思想，IoC意味着将设计好的对象交给容器进行控制，而不是直接在对象内部进行控制，以降低模块间的耦合度。IoC容器掌控创建和查找依赖对象的权限，所以对象与对象之间是松散耦合，测试方便，有利于功能复用，使程序结构更加灵活。

IoC很好的体现了面向对象设计法则之一—— 好莱坞法则：“别找我们，我们找你”；即由IoC容器帮对象找相应的依赖对象并注入，而不是由对象主动去找。

“控制（Control）”

在传统Java SE 程序设计中，要控制一个对象，通常是在对象内部通过new关键字进行对象的创建，然后由程序去创建和依赖对象，进行控制。在IoC中，一般是由IoC容器来进行对象的创建和控制。IoC主要对外部资源获取进行控制（不止对象，包括文件等）。

“反转（Inversion）“

在传统Java SE 程序设计中，需要自己在对象中主动获取依赖对象，即可理解为“正转”。在IoC中，是容器去查找、注入依赖的对象，然后获取依赖对象，依赖对象的获取方式反过来了，即为“反转”。

依赖注入（DI）

DI—Dependency Injection，即“依赖注入”，指由容器动态的将某个依赖关系注入到组件之中。

理解DI的关键是：“谁依赖谁，为什么需要依赖，谁注入谁，注入了什么”，那我们来深入分析一下：

●谁依赖于谁：当然是应用程序依赖于IoC容器；

●为什么需要依赖：应用程序需要IoC容器来提供对象需要的外部资源；

●谁注入谁：很明显是IoC容器注入应用程序某个对象，应用程序依赖的对象；

●注入了什么：就是注入某个对象所需要的外部资源（包括对象、资源、常量数据）。

IoC和DI由什么关系呢？其实它们是同一个概念的不同角度描述，由于控制反转概念比较含糊（可能只是理解为容器控制对象这一个层面，很难让人想到谁来维护对象关系），所以2004年大师级人物Martin Fowler又给出了一个新的名字：“依赖注入”，相对IoC 而言，“依赖注入”明确描述了“被注入对象依赖IoC容器配置依赖对象”。

区别、特点

设值注入优点

  • 与传统的JavaBean的写法更相似，程序开发人员更容易理解、接受。通过setter方法设定依赖关系显得更加直观、自然。
  • 对于复杂的依赖关系，如果采用构造注入，会导致构造器过于臃肿，难以阅读。Spring在创建Bean实例时，需要同时实例化其依赖的全部实例，因而导致性能下降。而使用设值注入，则能避免这些问题。
  • 尤其是在某些属性可选的情况下，多参数的构造器更加笨重。

设值注入的缺点

在设值注入中，需要对每个需要注入的类的属性定义public setter方法，因此可以对这个类内的属性进行选择性注入，这种方法虽然有很大的灵活性，但容易破坏类的状态和行为。如对于同一类内的两个属性a和b，只对a进行了注入，但之后调用方法时需要调用a和b的值，调用后的结果可能会出错

构造注入优点

  • 构造注入很好的体现了Java的设计原则之一：”在构造期间即创建一个完整，合法的对象“。

  • 构造注入可以在构造器中决定依赖关系的注入顺序，优先依赖的优先注入。例如，组件中其他依赖关系的注入，常常需要依赖于Datasource的注入。采用构造注入，可以在代码中清晰地决定注入顺序。

  • 对于依赖关系无须变化的Bean，构造注入更有用处。因为没有setter方法，所有的依赖关系全部在构造器内设定。因此，无须担心后续的代码对依赖关系产生破坏。

  • 依赖关系只能在构造器中设定，则只有组件的创建者才能改变组件的依赖关系。对组件的调用者而言，组件内部的依赖关系完全透明，更符合高内聚的原则。

构造注入的缺点

  • 构造方法里如出现错误，有时比较难查。
  • 在继承中，构造方法不会自动继承下去，必须手动完成。
  • 对于一些特殊情况，该方法很难进行处理。比如对于不是必须的而是可选择的属性（部分原因是Java不能在构造方法中设定默认值），或是循环的依赖关系等。

总结

综上，DI就是将协助对象之间合作的任务从对象本身转移出来，而由Spring框架来进行负责。IoC是由容器来控制程序间的关系，相对于由程序之间控制的方法，控制权发生了反转。
